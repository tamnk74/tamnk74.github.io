SCM Policy for NestJS & Next.js Microservices

Overview​

This guide provides a comprehensive Source Control Management (SCM) policy specifically designed for NestJS microservices and Next.js microfrontend architectures.

Maturity Level: 1 (Foundation)​

Area: Architecture Definition
Practice: Architecture & Coding Guidance
Phase: SETUP

Branch Strategy for Microservices Architecture​

Main Branches​

gitGraph:
    commit id: "Initial"
    branch develop
    checkout develop
    commit id: "Feature Integration"
    branch feature/user-service
    checkout feature/user-service
    commit id: "Add Auth"
    checkout develop
    merge feature/user-service
    checkout main
    merge develop
    commit id: "Release v1.0"

Production Branches​

  • main - Production-ready code, protected branch
  • develop - Integration branch for next release

Supporting Branches​

  • feature/* - New features (e.g., feature/user-authentication)
  • hotfix/* - Critical production fixes (e.g., hotfix/payment-bug)
  • release/* - Release preparation (e.g., release/v1.2.0)

Microservice-Specific Branching​

For a microservices architecture with separate repositories:

ecommerce-platform/
├── user-service/          (NestJS microservice)
├── product-service/       (NestJS microservice)
├── order-service/         (NestJS microservice)
├── admin-frontend/        (Next.js microfrontend)
├── customer-frontend/     (Next.js microfrontend)
└── shared-components/     (Shared UI library)

Each service follows the same branching strategy independently.

Naming Conventions​

Branch Naming​

# Feature branches
feature/JIRA-123-user-authentication
feature/add-payment-gateway
feature/order-tracking

# Hotfix branches  
hotfix/JIRA-456-payment-timeout
hotfix/memory-leak-fix

# Release branches
release/v1.2.0
release/v2.0.0-beta

Commit Message Convention​

Use Conventional Commits format:

# Format
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]

# Examples for NestJS Microservices
feat(user-service): add JWT authentication middleware
fix(order-service): resolve inventory race condition
docs(api): update OpenAPI specifications
test(payment): add integration tests for refund flow

# Examples for Next.js Microfrontends
feat(checkout): implement payment form validation
fix(product-catalog): resolve image loading issue
style(admin-panel): update dashboard layout
perf(search): optimize product search performance

Commit Types​

  • feat: New feature
  • fix: Bug fix
  • docs: Documentation changes
  • style: Code style changes (formatting, etc.)
  • refactor: Code refactoring
  • perf: Performance improvements
  • test: Adding or updating tests
  • chore: Build process or auxiliary tool changes

Pull Request Guidelines​

PR Template for Microservices​

## Description
Brief description of changes

## Type of Change
- [ ] Bug fix (non-breaking change which fixes an issue)
- [ ] New feature (non-breaking change which adds functionality)
- [ ] Breaking change (fix or feature that would cause existing functionality to not work as expected)
- [ ] Documentation update

## Microservice Impact
- [ ] User Service
- [ ] Product Service  
- [ ] Order Service
- [ ] Payment Service

## Microfrontend Impact
- [ ] Admin Frontend
- [ ] Customer Frontend
- [ ] Mobile App

## Testing
- [ ] Unit tests pass
- [ ] Integration tests pass
- [ ] E2E tests pass
- [ ] Performance tests (if applicable)

## API Changes
- [ ] OpenAPI spec updated
- [ ] Breaking changes documented
- [ ] Backward compatibility maintained

## Deployment
- [ ] Database migrations included
- [ ] Environment variables documented
- [ ] Configuration changes noted

## Screenshots (if applicable)

## Related Issues
Closes #123

Review Requirements​

For NestJS Microservices​

  • Minimum 2 reviewers (1 senior developer + 1 architect)
  • Required checks:
    • Unit tests (coverage > 80%)
    • Integration tests
    • API contract validation
    • Security scan (Snyk, SonarQube)
    • Performance impact assessment

For Next.js Microfrontends​

  • Minimum 2 reviewers (1 frontend developer + 1 UX reviewer)
  • Required checks:
    • Component tests (React Testing Library)
    • Visual regression tests (Chromatic/Percy)
    • Accessibility checks (axe-core)
    • Bundle size analysis
    • Cross-browser compatibility

Merge Strategies​

Feature Integration​

# Squash and merge for clean history
git checkout develop
git merge --squash feature/user-authentication
git commit -m "feat(user-service): add JWT authentication

- Implement JWT middleware
- Add user role-based access control  
- Update OpenAPI documentation
- Add integration tests

Closes #123"

Release Process​

# Create release branch
git checkout develop
git checkout -b release/v1.2.0

# Finalize release
git checkout main
git merge --no-ff release/v1.2.0
git tag -a v1.2.0 -m "Release version 1.2.0"

# Merge back to develop
git checkout develop  
git merge --no-ff release/v1.2.0

Microservice Coordination​

Cross-Service Dependencies​

When changes affect multiple services:

  1. Create coordination PR in infrastructure repository
  2. Update service contracts in shared schema repository
  3. Coordinate deployment order through release planning
  4. Use feature flags for gradual rollout

Example: Order Service + Payment Service Integration​

# 1. Update shared contracts
git checkout -b feature/payment-integration contracts-repo

# 2. Implement in order service
git checkout -b feature/payment-integration order-service

# 3. Implement in payment service  
git checkout -b feature/payment-integration payment-service

# 4. Update frontend
git checkout -b feature/payment-integration checkout-frontend

Microfrontend Coordination​

Module Federation Updates​

For Next.js microfrontends using Module Federation:

# Shell application
git checkout -b feature/add-user-dashboard shell-app

# User microfrontend
git checkout -b feature/user-dashboard user-mf

# Coordinate through package.json updates
# and module federation configuration

Environment-Specific Configurations​

Development Environment​

  • Branch protection: Disabled for rapid iteration
  • Auto-merge: Enabled for approved PRs
  • Deployment: Automatic on merge to develop

Staging Environment​

  • Branch protection: Required status checks
  • Manual approval: Required for releases
  • Deployment: Automatic on release branch creation

Production Environment​

  • Branch protection: Strict (no direct pushes)
  • Multiple approvals: Required (2+ senior developers)
  • Deployment: Manual trigger only
  • Rollback plan: Automated via tags

Implementation Checklist​

Setup Phase​

  • Create branch protection rules
  • Configure automated testing
  • Set up code quality gates
  • Configure deployment pipelines
  • Create PR templates
  • Document rollback procedures

NestJS Microservice Specific​

  • Configure API versioning strategy
  • Set up service mesh integration
  • Configure distributed tracing
  • Set up health check endpoints
  • Configure metrics collection

Next.js Microfrontend Specific​

  • Configure Module Federation
  • Set up shared component library
  • Configure bundle analysis
  • Set up visual regression testing
  • Configure performance monitoring

Tools and Automation​

Required Tools​

{
  "husky": "^8.0.0",
  "lint-staged": "^13.0.0",
  "@commitlint/cli": "^17.0.0",
  "@commitlint/config-conventional": "^17.0.0"
}

Git Hooks Configuration​

// .husky/pre-commit
npx lint-staged

// .husky/commit-msg  
npx --no -- commitlint --edit ${1}

Lint-staged Configuration​

{
  "lint-staged": {
    "*.{ts,tsx}": ["eslint --fix", "prettier --write"],
    "*.{json,md}": ["prettier --write"],
    "*.{css,scss}": ["stylelint --fix"]
  }
}

Monitoring and Metrics​

SCM Health Metrics​

  • PR merge time (target: < 24 hours)
  • Build success rate (target: > 95%)
  • Code review coverage (target: 100%)
  • Branch age (target: < 7 days)

Microservice Metrics​

  • Service deployment frequency
  • Cross-service dependency changes
  • API contract violations
  • Rollback frequency

Troubleshooting Common Issues​

Merge Conflicts in Microservices​

# Use automated conflict resolution
git config merge.ours.driver true
git config merge.npm.driver "npm install"

Coordinating Frontend-Backend Changes​

# Use feature flags for gradual rollout
# Frontend checks feature flag before using new API
if (featureFlags.newPaymentFlow) {
  // Use new payment API
} else {
  // Use legacy payment API  
}
