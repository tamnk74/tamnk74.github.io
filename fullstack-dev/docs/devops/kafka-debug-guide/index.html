Kafka Debug and Troubleshooting Guide

Overview​

This guide provides comprehensive debugging steps and troubleshooting procedures for Apache Kafka deployment on Google Kubernetes Engine (GKE) for the e-commerce microservices platform. This document covers all the issues encountered and solutions implemented during the Kafka setup process.

Quick Debug Commands​

1. Cluster Connection​

# Connect to GKE cluster
gcloud container clusters get-credentials sea1-prod-ecom-gke \
  --zone asia-southeast1-a \
  --project one-global-ecommerce-prod

# Verify connection
kubectl cluster-info
kubectl get nodes

2. Kafka Status Check​

# Check all Kafka resources
kubectl get all -n kafka

# Check pods status
kubectl get pods -n kafka -o wide

# Check services and endpoints
kubectl get services -n kafka
kubectl get endpoints -n kafka

# Check persistent volumes
kubectl get pv,pvc -n kafka

3. Quick Health Check​

# Check if Zookeeper is responding
kubectl exec -n kafka deployment/zookeeper -- echo srvr | nc localhost 2181

# Check if Kafka is responding
kubectl exec -n kafka deployment/kafka-broker -- kafka-topics --bootstrap-server localhost:9092 --list

# Check topic creation capability
kubectl exec -n kafka deployment/kafka-broker -- kafka-topics --bootstrap-server localhost:9092 --create --topic test-topic --partitions 1 --replication-factor 1 --if-not-exists

Common Issues and Solutions​

Issue 1: Mount Path Permission Errors​

Error: Command [/usr/local/bin/dub path /var/lib/zookeeper/data writable] FAILED

Root Cause: Zookeeper container running as non-root user cannot write to mounted volumes.

Solution:

# Add fsGroup to deployment security context
spec:
  template:
    spec:
      securityContext:
        fsGroup: 1000
        runAsUser: 1000
        runAsNonRoot: true

Debug Commands:

# Check pod security context
kubectl describe pod -n kafka <zookeeper-pod-name>

# Check volume permissions
kubectl exec -n kafka <zookeeper-pod-name> -- ls -la /var/lib/zookeeper/

# Test write permissions
kubectl exec -n kafka <zookeeper-pod-name> -- touch /var/lib/zookeeper/data/test-file

Issue 2: Lost+Found Directory Error​

Error: Found directory /var/lib/kafka/data/lost+found

Root Cause: Kafka rejects any unexpected directories in its data directory, including filesystem-generated lost+found.

Solution:

# Change KAFKA_LOG_DIRS to use subdirectory
- name: KAFKA_LOG_DIRS
  value: '/var/lib/kafka/data/kafka-logs' # Instead of '/var/lib/kafka/data'

Debug Commands:

# Check Kafka data directory contents
kubectl exec -n kafka <kafka-pod-name> -- ls -la /var/lib/kafka/data/

# Check Kafka logs for directory errors
kubectl logs -n kafka deployment/kafka-broker | grep "Found directory"

# Verify log directory configuration
kubectl exec -n kafka <kafka-pod-name> -- env | grep KAFKA_LOG_DIRS

Issue 3: DNS Resolution Errors (Cross-Namespace)​

Error: KafkaJSConnectionError: Connection error: getaddrinfo ENOTFOUND kafka-service

Root Cause: Application running in different namespace cannot resolve kafka-service without FQDN.

Solution:

# Create ExternalName service in application namespace
apiVersion: v1
kind: Service
metadata:
  name: kafka-service
  namespace: prod-env # Application namespace
spec:
  type: ExternalName
  externalName: kafka-service.kafka.svc.cluster.local
  ports:
    - port: 9092

Debug Commands:

# Check which namespace your application is running in
kubectl get pods --all-namespaces | grep -E "(app|backend|frontend|ecom)"

# Test DNS resolution from application namespace
kubectl run -n prod-env test-pod --image=busybox --rm -it -- nslookup kafka-service

# Test connection from application namespace
kubectl run -n prod-env test-pod --image=busybox --rm -it -- telnet kafka-service 9092

# Check service endpoints
kubectl get endpoints -n kafka kafka-service
kubectl get endpoints -n prod-env kafka-service

Issue 4: Topic Not Found Errors​

Error: UNKNOWN_TOPIC_OR_PARTITION with error code 3

Root Cause: Topics don't exist and auto-creation is disabled.

Solution:

# Option 1: Enable auto-creation (development)
# Set KAFKA_AUTO_CREATE_TOPICS_ENABLE: 'true'

# Option 2: Create topics manually (production)
kubectl apply -f kafka-topics-setup.yaml

Debug Commands:

# List existing topics
kubectl exec -n kafka deployment/kafka-broker -- kafka-topics --bootstrap-server localhost:9092 --list

# Check topic details
kubectl exec -n kafka deployment/kafka-broker -- kafka-topics --bootstrap-server localhost:9092 --describe

# Check auto-creation setting
kubectl exec -n kafka deployment/kafka-broker -- env | grep KAFKA_AUTO_CREATE_TOPICS_ENABLE

# Test topic creation
kubectl exec -n kafka deployment/kafka-broker -- kafka-topics --bootstrap-server localhost:9092 --create --topic test-topic --partitions 3 --replication-factor 1

Issue 5: Security Policy Violations​

Error: Pod security policy violations for non-root containers

Root Cause: Init containers or main containers trying to run as root.

Solution:

# Ensure all containers run as non-root
securityContext:
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: false
  runAsNonRoot: true
  runAsUser: 1000
  runAsGroup: 1000
  capabilities:
    drop:
      - ALL

Debug Commands:

# Check pod security context
kubectl describe pod -n kafka <pod-name> | grep -A 20 "Security Context"

# Check for security policy violations
kubectl get events -n kafka | grep -i security

# Verify user/group IDs
kubectl exec -n kafka <pod-name> -- id

Deployment Troubleshooting​

1. Pod Startup Issues​

# Check pod events for startup issues
kubectl describe pod -n kafka <pod-name>

# Check init container logs
kubectl logs -n kafka <pod-name> -c <init-container-name>

# Check main container logs
kubectl logs -n kafka <pod-name> -c <container-name>

# Check resource constraints
kubectl top pods -n kafka
kubectl describe nodes | grep -A 5 "Allocated resources"

2. Persistent Volume Issues​

# Check PVC status
kubectl get pvc -n kafka

# Check PV binding
kubectl get pv | grep kafka

# Check storage class
kubectl get storageclass

# Check volume mount issues
kubectl describe pod -n kafka <pod-name> | grep -A 10 "Mounts"

3. Service and Network Issues​

# Check service configuration
kubectl get svc -n kafka -o yaml

# Test service connectivity
kubectl run test-pod --image=busybox --rm -it -- telnet kafka-service.kafka.svc.cluster.local 9092

# Check endpoints
kubectl get endpoints -n kafka

# Check network policies
kubectl get networkpolicy -n kafka

Network and DNS Issues​

Cross-Namespace Communication​

# Test DNS resolution across namespaces
kubectl run -n prod-env dns-test --image=busybox --rm -it -- nslookup kafka-service.kafka.svc.cluster.local

# Test connectivity from different namespaces
kubectl run -n prod-env net-test --image=busybox --rm -it -- telnet kafka-service.kafka.svc.cluster.local 9092

# Check service discovery
kubectl get services --all-namespaces | grep kafka

Service Mesh Integration​

# Check if Istio is installed and affecting traffic
kubectl get pods -n istio-system

# Check Istio configuration for Kafka
kubectl get destinationrule -n kafka
kubectl get virtualservice -n kafka

# Check Istio sidecar injection
kubectl get pods -n kafka -o yaml | grep istio

Security Context Problems​

1. Non-Root User Setup​

# Verify container user context
kubectl exec -n kafka <pod-name> -- whoami
kubectl exec -n kafka <pod-name> -- id

# Check file permissions
kubectl exec -n kafka <pod-name> -- ls -la /var/lib/kafka/
kubectl exec -n kafka <pod-name> -- ls -la /var/lib/zookeeper/

# Test write permissions
kubectl exec -n kafka <pod-name> -- touch /var/lib/kafka/data/test-write

2. Volume Permission Fixes​

# Check fsGroup application
kubectl describe pod -n kafka <pod-name> | grep fsGroup

# Verify volume ownership
kubectl exec -n kafka <pod-name> -- ls -ldn /var/lib/kafka/data/

# Manual permission fix (if needed)
kubectl exec -n kafka <pod-name> -- chown -R 1000:1000 /var/lib/kafka/data/

Volume and Storage Issues​

1. PVC Troubleshooting​

# Check PVC events
kubectl describe pvc -n kafka <pvc-name>

# Check storage provisioner
kubectl get storageclass -o yaml

# Check available storage
kubectl get pv

# Force PVC deletion (if stuck)
kubectl patch pvc -n kafka <pvc-name> -p '{"metadata":{"finalizers":null}}'

2. Data Recovery​

# Backup existing data before changes
kubectl exec -n kafka <pod-name> -- tar -czf /tmp/kafka-backup.tar.gz /var/lib/kafka/data/

# Copy backup to local machine
kubectl cp kafka/<pod-name>:/tmp/kafka-backup.tar.gz ./kafka-backup.tar.gz

# Restore data (if needed)
kubectl cp ./kafka-backup.tar.gz kafka/<pod-name>:/tmp/
kubectl exec -n kafka <pod-name> -- tar -xzf /tmp/kafka-backup.tar.gz -C /

Topic Management​

1. Topic Creation and Management​

# Create topics with specific configuration
kubectl exec -n kafka deployment/kafka-broker -- kafka-topics \
  --bootstrap-server localhost:9092 \
  --create \
  --topic order-events \
  --partitions 3 \
  --replication-factor 1 \
  --config retention.ms=604800000

# List all topics
kubectl exec -n kafka deployment/kafka-broker -- kafka-topics \
  --bootstrap-server localhost:9092 \
  --list

# Describe topic configuration
kubectl exec -n kafka deployment/kafka-broker -- kafka-topics \
  --bootstrap-server localhost:9092 \
  --describe \
  --topic order-events

# Delete topic (if needed)
kubectl exec -n kafka deployment/kafka-broker -- kafka-topics \
  --bootstrap-server localhost:9092 \
  --delete \
  --topic test-topic

2. Topic Configuration Updates​

# Update topic configuration
kubectl exec -n kafka deployment/kafka-broker -- kafka-configs \
  --bootstrap-server localhost:9092 \
  --entity-type topics \
  --entity-name order-events \
  --alter \
  --add-config retention.ms=1209600000

# View topic configuration
kubectl exec -n kafka deployment/kafka-broker -- kafka-configs \
  --bootstrap-server localhost:9092 \
  --entity-type topics \
  --entity-name order-events \
  --describe

Application Connection Issues​

1. KafkaJS Connection Debugging​

# Test connection from application pod
kubectl exec -n prod-env <app-pod> -- node -e "
const { Kafka } = require('kafkajs');
const kafka = Kafka({
  clientId: 'debug-client',
  brokers: ['kafka-service:9092']
});
kafka.admin().connect().then(() => console.log('Connected')).catch(console.error);
"

# Check if topics are accessible from application
kubectl run -n prod-env kafka-test --image=confluentinc/cp-kafka:7.4.0 --rm -it -- \
  kafka-topics --bootstrap-server kafka-service:9092 --list

2. Producer/Consumer Testing​

# Test producer
kubectl run -n prod-env kafka-producer --image=confluentinc/cp-kafka:7.4.0 --rm -it -- \
  kafka-console-producer --bootstrap-server kafka-service:9092 --topic test-topic

# Test consumer
kubectl run -n prod-env kafka-consumer --image=confluentinc/cp-kafka:7.4.0 --rm -it -- \
  kafka-console-consumer --bootstrap-server kafka-service:9092 --topic test-topic --from-beginning

Performance and Monitoring​

1. Resource Monitoring​

# Check resource usage
kubectl top pods -n kafka

# Check memory and CPU usage over time
kubectl logs -n kafka deployment/kafka-broker | grep -E "(memory|cpu|heap)"

# Monitor disk usage
kubectl exec -n kafka <kafka-pod> -- df -h /var/lib/kafka/data

2. Performance Tuning​

# Check Kafka JVM settings
kubectl exec -n kafka deployment/kafka-broker -- env | grep KAFKA_HEAP_OPTS

# Monitor broker metrics
kubectl exec -n kafka deployment/kafka-broker -- kafka-broker-api-versions --bootstrap-server localhost:9092

# Check broker configuration
kubectl exec -n kafka deployment/kafka-broker -- kafka-configs \
  --bootstrap-server localhost:9092 \
  --entity-type brokers \
  --entity-name 1 \
  --describe

3. Log Analysis​

# Monitor real-time logs
kubectl logs -n kafka deployment/kafka-broker -f

# Search for specific errors
kubectl logs -n kafka deployment/kafka-broker | grep -E "(ERROR|WARN|Exception)"

# Export logs for analysis
kubectl logs -n kafka deployment/kafka-broker > kafka-broker.log
kubectl logs -n kafka deployment/zookeeper > zookeeper.log

Recovery Procedures​

1. Complete Kafka Reset​

# Use the cleanup script
chmod +x cleanup-and-deploy-kafka.sh
./cleanup-and-deploy-kafka.sh

2. Rolling Restart​

# Restart Zookeeper first
kubectl rollout restart deployment/zookeeper -n kafka
kubectl rollout status deployment/zookeeper -n kafka

# Then restart Kafka
kubectl rollout restart deployment/kafka-broker -n kafka
kubectl rollout status deployment/kafka-broker -n kafka

3. Scale Down and Up​

# Scale down to zero
kubectl scale deployment kafka-broker --replicas=0 -n kafka
kubectl scale deployment zookeeper --replicas=0 -n kafka

# Wait for pods to terminate
kubectl get pods -n kafka

# Scale back up
kubectl scale deployment zookeeper --replicas=1 -n kafka
kubectl wait --for=condition=available --timeout=300s deployment/zookeeper -n kafka

kubectl scale deployment kafka-broker --replicas=1 -n kafka
kubectl wait --for=condition=available --timeout=300s deployment/kafka-broker -n kafka

4. Emergency Data Backup​

# Create emergency backup of Kafka data
kubectl exec -n kafka <kafka-pod> -- tar -czf /tmp/emergency-backup.tar.gz /var/lib/kafka/data/

# Copy to safe location
kubectl cp kafka/<kafka-pod>:/tmp/emergency-backup.tar.gz ./emergency-backup-$(date +%Y%m%d-%H%M%S).tar.gz

# Backup Zookeeper data
kubectl exec -n kafka <zookeeper-pod> -- tar -czf /tmp/zk-backup.tar.gz /var/lib/zookeeper/data/
kubectl cp kafka/<zookeeper-pod>:/tmp/zk-backup.tar.gz ./zk-backup-$(date +%Y%m%d-%H%M%S).tar.gz

# Topic setup configuration
cat kafka-topics-setup.yaml

cat kafka-service-prod-env.yaml

# Deployment automation
cat cleanup-and-deploy-kafka.sh

Best Practices Learned​

  1. Always use fsGroup: Set fsGroup: 1000 for volume permission handling
  2. Subdirectories for data: Use subdirectories like /var/lib/kafka/data/kafka-logs to avoid filesystem conflicts
  3. Cross-namespace services: Create ExternalName services for cross-namespace communication
  4. Topic management: Use dedicated jobs for topic creation in production
  5. Resource limits: Always set appropriate resource requests and limits
  6. Health checks: Implement proper startup and liveness probes
  7. Security contexts: Follow non-root container best practices
  8. Monitoring: Implement comprehensive logging and monitoring
