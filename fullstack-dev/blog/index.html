Elasticsearch Full-Text Search Setup

October 1, 2025 · 5 min read

This guide will help you set up Elasticsearch for full-text search functionality in your Next.js application.

Prerequisites​

  • Node.js and npm/yarn installed
  • Next.js application running
  • Supabase database with posts

1. Install Elasticsearch​

Option A: Using Docker (Recommended for development)​

# Start Elasticsearch with Docker
docker run -d \
  --name elasticsearch \
  -p 9200:9200 \
  -e "discovery.type=single-node" \
  -e "xpack.security.enabled=false" \
  elasticsearch:8.11.0

# Verify Elasticsearch is running
curl http://localhost:9200

Option B: Local Installation​

  1. Download Elasticsearch from elastic.co
  2. Follow the installation guide for your operating system
  3. Start Elasticsearch service

2. Environment Variables​

Add these environment variables to your .env.local file:

# Elasticsearch Configuration
ELASTICSEARCH_URL=http://localhost:9200
ELASTICSEARCH_POSTS_INDEX=posts

# Optional: Authentication (if your Elasticsearch requires auth)
# ELASTICSEARCH_USERNAME=your_username
# ELASTICSEARCH_PASSWORD=your_password
# ELASTICSEARCH_APIKEY=your_api_key

3. Initialize the Index​

Option A: Using the Web Interface (Development)​

  1. Go to your posts page (/posts)
  2. In the debug section, click "Check Status" to verify Elasticsearch connection
  3. Click "Initialize Index" to create the index and bulk import existing posts

Option B: Using the Setup Script​

# Run the setup script
npx tsx scripts/setup-elasticsearch.ts

# Or add to package.json scripts:
# "setup:elasticsearch": "tsx scripts/setup-elasticsearch.ts"
# Then run: npm run setup:elasticsearch

Option C: Using the API Endpoint​

# Check Elasticsearch status
curl http://localhost:3000/api/elasticsearch/init

# Initialize the index
curl -X POST http://localhost:3000/api/elasticsearch/init

4. Features​

Search Functionality​

  • Full-text search: Search across post titles and content
  • Fuzzy matching: Handles typos and similar words
  • Highlighting: Search terms are highlighted in results
  • Fallback: Falls back to database search if Elasticsearch is unavailable
  • Responsive: Search bar available in navbar on desktop and mobile

Search Features​

  • Keyboard shortcut: Press Cmd+K (Mac) or Ctrl+K (Windows/Linux) to focus search
  • Real-time results: Search results update as you type
  • Pagination: Large result sets are paginated
  • Sorting: Results sorted by relevance score and creation date

5. API Endpoints​

Search Posts​

GET /api/posts/search?q=search_term&page=1&limit=10

Initialize Elasticsearch​

GET /api/elasticsearch/init    # Check status
POST /api/elasticsearch/init   # Initialize index

Sync Posts to Elasticsearch​

GET /api/elasticsearch/sync     # Health check and endpoints info
POST /api/elasticsearch/sync?action=sync   # Sync all posts
POST /api/elasticsearch/sync?action=health # Health check only

6. Automatic Indexing​

Posts are automatically indexed in Elasticsearch when:

  • Creating a post: New posts are indexed immediately after successful creation
  • Updating a post: Modified posts are re-indexed with the latest content
  • Deleting a post: Posts are removed from the Elasticsearch index

The indexing happens automatically in the background and won't affect the user experience. If Elasticsearch is unavailable, the database operations will still succeed, and only a warning will be logged.

Manual Sync​

If you need to sync existing posts or re-index everything:

Using the API​

# Check Elasticsearch health
curl http://localhost:3000/api/elasticsearch/sync

# Sync all posts to Elasticsearch
curl -X POST "http://localhost:3000/api/elasticsearch/sync?action=sync"

# Health check only
curl -X POST "http://localhost:3000/api/elasticsearch/sync?action=health"

Using the Sync Service​

import { syncPostsToElasticsearch } from '@/shared/services/sync';

const result = await syncPostsToElasticsearch();
console.log(`Synced ${result.syncedCount} posts`);

Monitoring Indexing​

During development, you can monitor indexing in the browser console:

  • ✅ Post indexed in Elasticsearch: 123
  • ✅ Post updated in Elasticsearch: 123
  • ✅ Post deleted from Elasticsearch: 123
  • ⚠️ Failed to index post in Elasticsearch: [error details]

7. Troubleshooting​

Elasticsearch not connecting​

  1. Verify Elasticsearch is running: curl http://localhost:9200
  2. Check environment variables in .env.local
  3. Check Docker container status: docker ps
  4. View Docker logs: docker logs elasticsearch

Search not working​

  1. Check if index exists: Use the debug panel in /posts
  2. Verify posts are indexed: Look for "posts indexed" message
  3. Check browser console for JavaScript errors
  4. Verify API endpoints are working

Performance Issues​

  1. Increase Elasticsearch memory: Add -e "ES_JAVA_OPTS=-Xms512m -Xmx512m" to Docker command
  2. Optimize search queries in elasticsearch.ts
  3. Implement caching for frequently searched terms

8. Production Deployment​

Elasticsearch Service​

For production, consider using:

  • Elastic Cloud: Managed Elasticsearch service
  • AWS OpenSearch: Amazon's Elasticsearch service
  • Self-hosted: Deploy Elasticsearch on your own infrastructure

Environment Variables for Production​

ELASTICSEARCH_URL=https://your-production-elasticsearch-url
ELASTICSEARCH_APIKEY=your-production-api-key
ELASTICSEARCH_POSTS_INDEX=production_posts

Security Considerations​

  1. Enable authentication on Elasticsearch
  2. Use HTTPS for Elasticsearch connections
  3. Restrict network access to Elasticsearch
  4. Regularly update Elasticsearch version

9. Monitoring and Maintenance​

Health Checks​

The application includes health checks:

  • Elasticsearch availability check in search API
  • Fallback to database search if Elasticsearch is down
  • Status indicators in development debug panel

Index Maintenance​

  • Reindexing: Use the initialize endpoint to rebuild the index
  • Mapping updates: Update the mapping in elasticsearch.ts and reinitialize
  • Performance monitoring: Monitor query performance and optimize as needed

10. Customization​

Search Configuration​

Edit src/shared/services/elasticsearch.ts to customize:

  • Search fields and boost values
  • Highlighting settings
  • Fuzzy matching sensitivity
  • Result sorting criteria

UI Customization​

Edit components to customize:

  • src/shared/ui/search-bar/index.tsx: Search bar appearance
  • src/features/posts/list/ui.tsx: Results display
  • src/app/[locale]/components/Navbar.tsx: Navbar integration

Support​

If you encounter issues:

  1. Check the troubleshooting section above
  2. Review Elasticsearch logs
  3. Check browser console for errors
  4. Verify environment variables are set correctly

SEO Setup with next-sitemap

October 1, 2025 · 3 min read

This project uses next-sitemap to generate SEO-optimized sitemaps and robots.txt files.

Features​

✅ Automatic Sitemap Generation: Generates comprehensive sitemaps for all public routes
✅ Multi-language Support: Full support for English (en) and Vietnamese (vi) locales
✅ Robots.txt Generation: Automatic robots.txt with AI crawler blocking
✅ SEO Optimization: Proper priorities, change frequencies, and hreflang attributes
✅ Build Integration: Automatically runs after each build

Configuration​

Environment Variables​

Set your production domain in your environment:

# .env.local or .env.production
NEXT_PUBLIC_APP_URL=https://next-for-prod.com

Routes Included​

The sitemap includes these public routes for both locales:

  • Homepage: /en, /vi (Priority: 1.0, Daily updates)
  • Content Pages: /about, /libs, /posts, /calendar (High priority)
  • Tools: /tools, /search-*, /roadmap/* (Medium priority)
  • Auth Pages: /login, /signup (Lower priority)
  • Utility Pages: /feedback (Medium priority)

Routes Excluded​

These routes are excluded from search engines:

  • Admin pages (/admin/*)
  • User profiles (/profile/*, /me/*)
  • Private settings (/settings)
  • API routes (/api/*)
  • Test pages (/test-*)
  • Authentication callbacks (/auth/*)
  • Edit/create forms

Usage​

Generate Sitemap​

# Manual generation
pnpm sitemap:generate

# Automatic generation after build
pnpm build  # Runs next-sitemap via postbuild script

Generated Files​

  • public/sitemap.xml - Main sitemap with all URLs
  • public/robots.txt - Search engine directives

SEO Utilities​

The project includes SEO utilities in src/utils/seo.ts:

import { generatePageSEO } from '@/utils/seo';

export const metadata = generatePageSEO('home', 'en');

Available functions:

  • generateSEOMetadata() - Comprehensive metadata generation
  • generatePageSEO() - Page-specific SEO metadata
  • generateStructuredData() - JSON-LD structured data

AI Crawler Blocking​

The robots.txt automatically blocks these AI crawlers:

  • GPTBot
  • ChatGPT-User
  • CCBot
  • anthropic-ai
  • Claude-Web

Monitoring​

Google Search Console​

  1. Submit your sitemap: https://yourdomain.com/sitemap.xml
  2. Monitor indexing status
  3. Check for crawl errors

Validation Tools​

  • Google Rich Results Test
  • Bing Webmaster Tools
  • XML Sitemap Validator

Configuration File​

Edit next-sitemap.config.js to:

  • Add new routes
  • Modify priorities and change frequencies
  • Update hreflang mappings
  • Exclude additional paths

Best Practices​

  1. Update sitemap after major changes: Run pnpm sitemap:generate
  2. Monitor crawl budget: Use robots.txt to guide crawlers to important pages
  3. Regular audits: Check for 404s and crawl errors
  4. Performance: Exclude dynamic/user-specific pages from sitemap
  5. Localization: Ensure hreflang attributes are correct for international SEO

Troubleshooting​

Sitemap not updating​

  • Run pnpm build to regenerate
  • Check next-sitemap.config.js for syntax errors
  • Verify environment variables

Routes missing from sitemap​

  • Add to additionalPaths in config
  • Check exclusion rules
  • Ensure pages are statically accessible

robots.txt issues​

  • Verify generateRobotsTxt: true in config
  • Check policy rules for conflicts
  • Test with robots.txt validators

Social Login Setup (Google & GitHub)

October 1, 2025 · 3 min read

This document explains how to set up Google and GitHub OAuth providers in your Supabase project.

🔧 Supabase Configuration​

1. Google OAuth Setup​

  1. Create Google OAuth App:

    • Go to Google Cloud Console
    • Create a new project or select existing one
    • Enable Google+ API
    • Go to "Credentials" → "Create Credentials" → "OAuth 2.0 Client IDs"
    • Choose "Web application"
    • Add authorized redirect URIs: 
      https://your-project-ref.supabase.co/auth/v1/callback

  2. Configure in Supabase:

    • Go to your Supabase project dashboard
    • Navigate to "Authentication" → "Providers"
    • Enable "Google"
    • Add your Google Client ID and Client Secret
    • Save configuration

2. GitHub OAuth Setup​

  1. Create GitHub OAuth App:

    • Go to GitHub Settings → Developer settings → OAuth Apps
    • Click "New OAuth App"
    • Fill in the details:
      • Application name: Your app name
      • Homepage URL: Your app URL
      • Authorization callback URL: 
        https://your-project-ref.supabase.co/auth/v1/callback

  2. Configure in Supabase:

    • Go to your Supabase project dashboard
    • Navigate to "Authentication" → "Providers"
    • Enable "GitHub"
    • Add your GitHub Client ID and Client Secret
    • Save configuration

🚀 Implementation Features​

Frontend Components​

  • Login Form (src/features/auth/login-form/ui.tsx):

    • Google login button with brand colors
    • GitHub login button
    • Loading states and error handling
    • Disabled state during authentication

  • OAuth Callback (src/app/[locale]/auth/callback/route.ts):

    • Handles OAuth redirects from providers
    • Exchanges authorization codes for sessions
    • Redirects with success/error messages

User Experience​

  1. Login Flow:

    • User clicks Google/GitHub button
    • Redirected to provider's authorization page
    • After approval, redirected back to /auth/callback
    • Session established and redirected to home page

  2. Error Handling:

    • Provider authentication failures
    • Network errors
    • Invalid callback requests
    • User-friendly error messages

🔒 Security Features​

  • PKCE Flow: Uses Proof Key for Code Exchange for security
  • State Validation: Prevents CSRF attacks
  • Secure Redirects: Only allows authorized redirect URLs
  • Session Management: Proper cookie handling and session storage

🎨 UI/UX Improvements​

  • Brand Colors: Google (red) and GitHub (gray) themed buttons
  • Loading States: Buttons disabled during authentication
  • Clear Messaging: Success and error feedback
  • Responsive Design: Works on mobile and desktop

📝 Usage Notes​

  • Make sure to configure your OAuth apps with the correct callback URLs
  • Test both providers in development and production environments
  • The callback route handles both success and error cases
  • Users will be redirected to the home page after successful login

🐛 Troubleshooting​

  1. "Invalid redirect URI": Check your OAuth app callback URLs
  2. "Client ID not found": Verify Supabase provider configuration
  3. "Authentication failed": Check provider app permissions and status
  4. Session not persisting: Verify cookie settings and domain configuration
