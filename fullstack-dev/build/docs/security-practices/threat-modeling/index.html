Ecommerce Application Threat Modeling Document

Executive Summary​

This document provides a comprehensive threat model for our ecommerce platform built using a modern microservices architecture with Next.js microfrontends and NestJS backend services. The analysis identifies potential security threats, vulnerabilities, and recommended mitigations across all layers of the application stack.

Application Overview​

Technology Stack​

Frontend:

  • Next.js 14+ with App Router
  • React 18+ with TypeScript
  • Tailwind CSS for styling
  • Module Federation for microfrontends
  • Redux Toolkit for state management

Backend:

  • NestJS microservices architecture
  • TypeORM for database operations
  • JWT-based authentication
  • API Gateway pattern
  • Message queues (Kafka/RabbitMQ)

Infrastructure:

  • Google Kubernetes Engine (GKE)
  • Cloud SQL (PostgreSQL)
  • Cloud Storage & Filestore
  • Redis for caching
  • Elasticsearch for search

DevOps & Security:

  • GitHub Actions for CI/CD
  • ArgoCD for GitOps
  • OWASP ZAP for DAST
  • SonarQube for SAST
  • Datadog for monitoring

Business Context​

The ecommerce platform handles:

  • User registration and authentication
  • Product catalog management
  • Shopping cart and checkout processes
  • Payment processing
  • Order management
  • Inventory tracking
  • Customer support
  • Administrative functions

Trust Boundaries​

  1. Internet to DMZ: Public internet to load balancer/CDN
  2. DMZ to Application Layer: Load balancer to API Gateway
  3. Application to Service Layer: API Gateway to microservices
  4. Service to Data Layer: Microservices to databases
  5. Internal Service Communication: Inter-microservice communication
  6. Administrative Access: Admin interfaces and tools

Data Flow Analysis​

Critical Data Flows:​

  1. User Authentication Flow: User credentials → Auth Service → Identity Provider
  2. Product Browsing Flow: User request → API Gateway → Product Service → Database
  3. Checkout Flow: Cart data → Order Service → Payment Service → External Payment Gateway
  4. Administrative Flow: Admin user → Admin interface → Administrative services

OWASP Top 10 Mapping​

A01: Broken Access Control​

  • Components Affected: All microservices, Admin interfaces
  • Risks: Unauthorized data access, privilege escalation
  • Examples: Direct object references, missing authorization checks

A02: Cryptographic Failures​

  • Components Affected: Authentication service, Payment service
  • Risks: Data exposure, credential theft
  • Examples: Weak encryption, exposed secrets

A03: Injection​

  • Components Affected: Database layer, Search functionality
  • Risks: Data breach, system compromise
  • Examples: SQL injection, NoSQL injection, Command injection

A04: Insecure Design​

  • Components Affected: Overall architecture
  • Risks: Systemic vulnerabilities
  • Examples: Missing rate limiting, insufficient security controls

A05: Security Misconfiguration​

  • Components Affected: All infrastructure components
  • Risks: System compromise, data exposure
  • Examples: Default credentials, unnecessary services, verbose error messages

A06: Vulnerable Components​

  • Components Affected: All services using third-party dependencies
  • Risks: Known vulnerability exploitation
  • Examples: Outdated frameworks, vulnerable libraries

A07: Identification and Authentication Failures​

  • Components Affected: Authentication service, Session management
  • Risks: Account takeover, unauthorized access
  • Examples: Weak passwords, session fixation

A08: Software and Data Integrity Failures​

  • Components Affected: CI/CD pipeline, Package management
  • Risks: Supply chain attacks, compromised deployments
  • Examples: Compromised dependencies, unsigned packages

A09: Security Logging and Monitoring Failures​

  • Components Affected: All services
  • Risks: Delayed breach detection, compliance issues
  • Examples: Insufficient logging, no monitoring

A10: Server-Side Request Forgery (SSRF)​

  • Components Affected: Services making external requests
  • Risks: Internal network access, data exposure
  • Examples: Unvalidated external API calls

Mitigation Strategies​

High Priority Mitigations​

1. Enhanced Authentication Security​

Implementation Timeline: 2-4 weeks

// Enhanced JWT token validation
@Injectable()
export class JwtAuthGuard implements CanActivate {
  async canActivate(context: ExecutionContext): Promise<boolean> {
    const request = context.switchToHttp().getRequest();
    const token = this.extractTokenFromHeader(request);

    if (!token) {
      throw new UnauthorizedException();
    }

    try {
      // Validate token signature and expiration
      const payload = await this.jwtService.verifyAsync(token, {
        secret: process.env.JWT_SECRET,
        issuer: process.env.JWT_ISSUER,
        audience: process.env.JWT_AUDIENCE,
      });

      // Additional security checks
      await this.validateTokenClaims(payload);
      await this.checkTokenRevocation(payload.jti);

      request['user'] = payload;
    } catch {
      throw new UnauthorizedException();
    }
    return true;
  }
}

2. API Security Hardening​

Implementation Timeline: 3-6 weeks

// Rate limiting implementation
@Controller('api')
@UseGuards(ThrottlerGuard)
export class ApiController {
  @Throttle({ default: { limit: 100, ttl: 60000 } })
  @Post('sensitive-endpoint')
  async sensitiveOperation(@Body() data: SensitiveDto) {
    // Input validation
    await this.validateInput(data);
    // Business logic
    return await this.processRequest(data);
  }
}

// Input validation middleware
export class ValidationPipe implements PipeTransform {
  transform(value: any, metadata: ArgumentMetadata) {
    // Sanitize input
    const sanitized = this.sanitizeInput(value);
    // Validate against schema
    const validated = this.validateSchema(sanitized, metadata);
    return validated;
  }
}

3. Database Security Enhancement​

Implementation Timeline: 2-3 weeks

// Query parameterization
@Entity('users')
export class User {
  // Use parameterized queries
  static async findByEmail(email: string): Promise<User> {
    return await this.createQueryBuilder('user').where('user.email = :email', { email }).getOne();
  }
}

// Row-level security implementation
@Entity('orders')
export class Order {
  @CreateDateColumn()
  createdAt: Date;

  @Column()
  userId: string;

  // Ensure users can only access their own orders
  static async findUserOrders(userId: string): Promise<Order[]> {
    return await this.find({
      where: { userId },
      select: ['id', 'createdAt', 'status', 'total'],
    });
  }
}

4. Frontend Security Measures​

Implementation Timeline: 2-4 weeks

// Content Security Policy
const securityHeaders = {
  'Content-Security-Policy': [
    "default-src 'self'",
    "script-src 'self' 'unsafe-inline' https://trusted-cdn.com",
    "style-src 'self' 'unsafe-inline'",
    "img-src 'self' data: https:",
    "connect-src 'self' https://api.yourdomain.com",
    "frame-ancestors 'none'",
  ].join('; '),
  'X-Frame-Options': 'DENY',
  'X-Content-Type-Options': 'nosniff',
  'Referrer-Policy': 'strict-origin-when-cross-origin',
};

// Secure data handling
const SecureStorageService = {
  setItem: (key: string, value: any) => {
    // Encrypt sensitive data before storing
    const encrypted = encrypt(JSON.stringify(value));
    sessionStorage.setItem(key, encrypted);
  },

  getItem: (key: string) => {
    const encrypted = sessionStorage.getItem(key);
    if (encrypted) {
      return JSON.parse(decrypt(encrypted));
    }
    return null;
  },
};

ENTRYPOINT ["dumb-init", "--"]

3. Secrets Management​

Implementation Timeline: 3-4 weeks

// Secure configuration management
@Injectable()
export class ConfigService {
  private readonly config: Record<string, any>;

  constructor() {
    this.config = {
      database: {
        host: this.getSecret('DB_HOST'),
        password: this.getSecret('DB_PASSWORD'),
      },
      jwt: {
        secret: this.getSecret('JWT_SECRET'),
        expiresIn: process.env.JWT_EXPIRES_IN || '1h',
      },
      encryption: {
        algorithm: 'aes-256-gcm',
        key: this.getSecret('ENCRYPTION_KEY'),
      },
    };
  }

  private getSecret(key: string): string {
    // Integration with Google Secret Manager
    return process.env[key] || this.getFromSecretManager(key);
  }
}

Monitoring and Detection​

Security Monitoring Strategy​

1. Real-time Threat Detection​

// Anomaly detection service
@Injectable()
export class ThreatDetectionService {
  private readonly redis: Redis;

  async detectAnomalousLogin(userId: string, ip: string, userAgent: string) {
    const key = `login_pattern:${userId}`;
    const recentLogins = await this.redis.lrange(key, 0, 10);

    // Check for suspicious patterns
    if (this.isAnomalousLocation(ip, recentLogins)) {
      await this.triggerSecurityAlert('anomalous_location', { userId, ip });
    }

    if (this.isAnomalousDevice(userAgent, recentLogins)) {
      await this.triggerSecurityAlert('new_device', { userId, userAgent });
    }
  }

  async detectBruteForce(ip: string) {
    const attempts = await this.redis.incr(`failed_attempts:${ip}`);
    await this.redis.expire(`failed_attempts:${ip}`, 900); // 15 minutes

    if (attempts > 5) {
      await this.blockIP(ip, 3600); // 1 hour
      await this.triggerSecurityAlert('brute_force', { ip, attempts });
    }
  }
}

2. Security Metrics Dashboard​

// Security metrics collection
@Injectable()
export class SecurityMetricsService {
  private readonly metricsCollector: DatadogMetrics;

  recordAuthenticationEvent(success: boolean, method: string) {
    this.metricsCollector.increment('auth.attempts', 1, {
      success: success.toString(),
      method,
    });
  }

  recordAPICall(endpoint: string, statusCode: number, responseTime: number) {
    this.metricsCollector.increment('api.calls', 1, {
      endpoint,
      status: statusCode.toString(),
    });

    this.metricsCollector.histogram('api.response_time', responseTime, {
      endpoint,
    });
  }
}

Incident Response​

Security Incident Response Plan​

1. Incident Classification​

  • P0 (Critical): Active breach, data exposure, system compromise
  • P1 (High): Potential breach, security control bypass
  • P2 (Medium): Policy violations, suspicious activity
  • P3 (Low): Security warnings, minor violations

2. Response Procedures​

// Automated incident response
@Injectable()
export class IncidentResponseService {
  async handleSecurityIncident(incident: SecurityIncident) {
    // Immediate containment
    if (incident.severity === 'critical') {
      await this.emergencyContainment(incident);
    }

    // Notification
    await this.notifySecurityTeam(incident);

    // Evidence collection
    await this.collectEvidence(incident);

    // Remediation
    await this.initiateRemediation(incident);
  }

  private async emergencyContainment(incident: SecurityIncident) {
    switch (incident.type) {
      case 'account_compromise':
        await this.disableUserAccount(incident.userId);
        break;
      case 'suspicious_admin_activity':
        await this.disableAdminSession(incident.sessionId);
        break;
      case 'data_breach':
        await this.enableEmergencyMode();
        break;
    }
  }
}

Recommendations​

Immediate Actions (0-30 days)​

  1. Implement Enhanced Authentication

    • Multi-factor authentication for all admin accounts
    • Token refresh mechanisms
    • Account lockout policies

  2. API Security Hardening

    • Implement comprehensive rate limiting
    • Add request validation middleware
    • Enable API security logging

  3. Database Security

    • Enable query logging and monitoring
    • Implement database activity monitoring
    • Review and update access controls

Short-term Actions (1-3 months)​

  1. Security Testing Integration

    • Integrate security tests into CI/CD pipeline
    • Automated vulnerability scanning
    • Regular penetration testing

  2. Monitoring Enhancement

    • Automated alerting rules
