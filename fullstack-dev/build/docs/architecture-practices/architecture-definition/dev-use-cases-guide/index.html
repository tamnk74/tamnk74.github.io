Development Use Cases Guide

This guide provides step-by-step instructions for special development scenarios commonly encountered when building NestJS microservices and Next.js microfrontends.

Overview​

Special development use cases require specific tools, configurations, and procedures that go beyond standard development workflows. This guide covers environment setup, custom tooling, debugging procedures, and troubleshooting for complex scenarios.

Environment Setup Use Cases​

Multi-Service Local Development​

Setting up multiple NestJS services and Next.js applications locally with proper service discovery and communication.

Prerequisites​

  • Docker and Docker Compose installed
  • Node.js 18+ and pnpm/npm
  • Git and development tools configured

Setup Steps​

  1. Clone and Setup Services
# Clone the monorepo
git clone <repository-url>
cd <project-name>

# Install dependencies for all services
pnpm install

# Start infrastructure services
docker-compose up -d postgres redis rabbitmq
  1. Configure Service Discovery
// shared/config/services.config.ts
export const ServiceConfig = {
  userService: {
    host: process.env.USER_SERVICE_HOST || 'localhost',
    port: process.env.USER_SERVICE_PORT || 3001,
  },
  orderService: {
    host: process.env.ORDER_SERVICE_HOST || 'localhost', 
    port: process.env.ORDER_SERVICE_PORT || 3002,
  },
  gateway: {
    host: process.env.GATEWAY_HOST || 'localhost',
    port: process.env.GATEWAY_PORT || 3000,
  }
};
  1. Environment Variable Configuration
# .env.local (for local development)
DATABASE_URL="postgresql://user:password@localhost:5432/microservices_db"
REDIS_URL="redis://localhost:6379"
RABBITMQ_URL="amqp://localhost:5672"

# Service-specific variables
USER_SERVICE_PORT=3001
ORDER_SERVICE_PORT=3002
GATEWAY_PORT=3000

# JWT secrets
JWT_SECRET="your-dev-jwt-secret"
JWT_REFRESH_SECRET="your-dev-refresh-secret"

Frontend Development with Backend Proxy​

Setting up Next.js applications to proxy API calls to local NestJS services during development.

Configuration Steps​

  1. Next.js Proxy Setup
// next.config.js
/** @type {import('next').NextConfig} */
const nextConfig = {
  async rewrites() {
    return [
      {
        source: '/api/users/:path*',
        destination: 'http://localhost:3001/users/:path*',
      },
      {
        source: '/api/orders/:path*', 
        destination: 'http://localhost:3002/orders/:path*',
      },
      {
        source: '/api/auth/:path*',
        destination: 'http://localhost:3000/auth/:path*',
      },
    ];
  },
};

module.exports = nextConfig;
  1. API Client Configuration
// lib/api-client.ts
import axios from 'axios';

const apiClient = axios.create({
  baseURL: process.env.NODE_ENV === 'development' 
    ? 'http://localhost:3100' // Next.js dev server
    : process.env.NEXT_PUBLIC_API_BASE_URL,
  timeout: 10000,
});

// Request interceptor for auth
apiClient.interceptors.request.use((config) => {
  const token = localStorage.getItem('accessToken');
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

Database Migration Tools​

Setting up and using database migration tools for schema management across services.

TypeORM Migration Setup​

  1. Migration Configuration
// src/database/database.config.ts
import { TypeOrmModuleOptions } from '@nestjs/typeorm';

export const databaseConfig: TypeOrmModuleOptions = {
  type: 'postgres',
  host: process.env.DB_HOST || 'localhost',
  port: parseInt(process.env.DB_PORT) || 5432,
  username: process.env.DB_USERNAME || 'postgres',
  password: process.env.DB_PASSWORD || 'password',
  database: process.env.DB_NAME || 'microservice_db',
  entities: ['dist/**/*.entity{.ts,.js}'],
  migrations: ['dist/migrations/*{.ts,.js}'],
  synchronize: false, // Never use in production
  logging: process.env.NODE_ENV === 'development',
};
  1. Migration Scripts
// package.json scripts
{
  "scripts": {
    "migration:generate": "typeorm-ts-node-commonjs migration:generate -d src/database/data-source.ts",
    "migration:run": "typeorm-ts-node-commonjs migration:run -d src/database/data-source.ts",
    "migration:revert": "typeorm-ts-node-commonjs migration:revert -d src/database/data-source.ts",
    "schema:drop": "typeorm-ts-node-commonjs schema:drop -d src/database/data-source.ts"
  }
}
  1. Cross-Service Migration Coordination
#!/bin/bash
# scripts/run-migrations.sh
echo "Running migrations for all services..."

services=("user-service" "order-service" "payment-service")

for service in "${services[@]}"; do
  echo "Running migrations for $service"
  cd "services/$service"
  npm run migration:run
  cd ../../
done

echo "All migrations completed"

Debugging Procedures​

Microservice Communication Debugging​

Debugging inter-service communication issues and message queue problems.

Service Health Checks​

// src/health/health.controller.ts
import { Controller, Get } from '@nestjs/common';
import { HealthCheck, HealthCheckService, TypeOrmHealthIndicator } from '@nestjs/terminus';

@Controller('health')
export class HealthController {
  constructor(
    private health: HealthCheckService,
    private db: TypeOrmHealthIndicator,
  ) {}

  @Get()
  @HealthCheck()
  check() {
    return this.health.check([
      () => this.db.pingCheck('database'),
      () => this.checkExternalServices(),
    ]);
  }

  private async checkExternalServices() {
    // Check dependent services
    const checks = await Promise.allSettled([
      fetch('http://user-service:3001/health'),
      fetch('http://order-service:3002/health'),
    ]);
    
    return {
      external_services: {
        status: checks.every(check => check.status === 'fulfilled') ? 'up' : 'down',
        checks: checks.map((check, index) => ({
          service: ['user-service', 'order-service'][index],
          status: check.status === 'fulfilled' ? 'up' : 'down',
        })),
      },
    };
  }
}

Message Queue Debugging​

// src/messaging/rabbitmq-debug.service.ts
import { Injectable, Logger } from '@nestjs/common';
import { RabbitMQModule, AmqpConnection } from '@golevelup/nestjs-rabbitmq';

@Injectable()
export class RabbitMQDebugService {
  private readonly logger = new Logger(RabbitMQDebugService.name);

  constructor(private readonly amqpConnection: AmqpConnection) {}

  async debugQueue(queueName: string) {
    try {
      const channel = this.amqpConnection.channel;
      const queueInfo = await channel.checkQueue(queueName);
      
      this.logger.log(`Queue ${queueName} info:`, {
        messageCount: queueInfo.messageCount,
        consumerCount: queueInfo.consumerCount,
      });
      
      return queueInfo;
    } catch (error) {
      this.logger.error(`Failed to check queue ${queueName}:`, error);
      throw error;
    }
  }

  async purgeQueue(queueName: string) {
    try {
      const channel = this.amqpConnection.channel;
      await channel.purgeQueue(queueName);
      this.logger.log(`Purged queue ${queueName}`);
    } catch (error) {
      this.logger.error(`Failed to purge queue ${queueName}:`, error);
      throw error;
    }
  }
}

Error Boundary for Debugging​

// components/ErrorBoundary.tsx
import React from 'react';

interface ErrorBoundaryState {
  hasError: boolean;
  error?: Error;
  errorInfo?: React.ErrorInfo;
}

export class ErrorBoundary extends React.Component<
  React.PropsWithChildren<{}>,
  ErrorBoundaryState
> {
  constructor(props: React.PropsWithChildren<{}>) {
    super(props);
    this.state = { hasError: false };
  }

  static getDerivedStateFromError(error: Error): ErrorBoundaryState {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error, errorInfo: React.ErrorInfo) {
    this.setState({ error, errorInfo });
    
    // Log to external service in production
    if (process.env.NODE_ENV === 'production') {
      console.error('Error caught by boundary:', error, errorInfo);
    }
  }

  render() {
    if (this.state.hasError) {
      return (
        <div className="error-boundary">
          <h2>Something went wrong.</h2>
          {process.env.NODE_ENV === 'development' && (
            <details style={{ whiteSpace: 'pre-wrap' }}>
              <summary>Error details</summary>
              {this.state.error && this.state.error.toString()}
              <br />
              {this.state.errorInfo.componentStack}
            </details>
          )}
        </div>
      );
    }

    return this.props.children;
  }
}

export GATEWAY_PORT=3000

CORS Issues in Development​

Handling CORS problems when frontend and backend run on different ports:

// src/main.ts (NestJS)
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  
  // Configure CORS for development
  app.enableCors({
    origin: [
      'http://localhost:3000', // Next.js dev server
      'http://localhost:3100', // Alternative port
      'http://localhost:4200', // Angular dev server if needed
    ],
    methods: ['GET', 'POST', 'PUT', 'DELETE', 'PATCH'],
    allowedHeaders: ['Content-Type', 'Authorization'],
    credentials: true,
  });

  await app.listen(process.env.PORT || 3001);
}
bootstrap();

Best Practices​

Environment Configuration​

  • Use different database names/ports for each environment
  • Never commit sensitive environment variables
  • Use .env.example files to document required variables
  • Validate environment variables at application startup

Service Dependencies​

  • Implement health checks for all dependencies
  • Use circuit breakers for external service calls
  • Implement retry logic with exponential backoff
  • Monitor service dependencies continuously

Development Workflow​

  • Use consistent port assignments across team
  • Implement hot reload for faster development cycles
  • Use shared configuration for consistent environments
  • Document special setup requirements clearly
