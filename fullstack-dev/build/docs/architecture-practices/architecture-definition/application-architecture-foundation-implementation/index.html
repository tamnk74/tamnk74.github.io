Application Architecture Foundation Implementation Guide

This guide provides step-by-step instructions for achieving the architecture maturity requirement: "Application architecture foundation is defined and documented".

Overview​

The application architecture foundation establishes the complete structural foundation of your application, including layers, frameworks, key mechanisms, and design patterns. This is a Maturity Level 1 - PREPARATION (Sprint 0) requirement that must be completed before development begins.

Success Criteria​

By the end of this implementation, you should have:

  • ✅ Complete architecture documentation with visual diagrams
  • ✅ Defined application layers and their responsibilities
  • ✅ Documented frameworks and technology choices
  • ✅ Established key mechanisms (logging, error handling, security)
  • ✅ Architecture review sessions completed with stakeholders
  • ✅ Implementation guidelines for development teams

Step 2.2: Define Application Layers​

Duration: 2-3 days

Actions:

  1. Layer Definition Documentation

    # docs/architecture/application-layers.md

    ## Frontend Layers (Next.js)

    ### 1. Presentation Layer

    - **Responsibility**: User interface and user interaction
    - **Components**: Pages, components, forms, layouts
    - **Technologies**: React, Tailwind CSS, Framer Motion

    ### 2. Application Layer

    - **Responsibility**: Application logic and state management
    - **Components**: Custom hooks, context providers, utilities
    - **Technologies**: React Context, Zustand, React Query

    ### 3. Infrastructure Layer

    - **Responsibility**: External service integration
    - **Components**: API clients, authentication, storage
    - **Technologies**: Axios, NextAuth, LocalStorage

    ## Backend Layers (NestJS)

    ### 1. Presentation Layer (Controllers)

    - **Responsibility**: HTTP request handling and response formatting
    - **Components**: Controllers, DTOs, validation pipes
    - **Technologies**: NestJS controllers, class-validator

    ### 2. Application Layer (Services)

    - **Responsibility**: Business logic orchestration
    - **Components**: Application services, use cases
    - **Technologies**: NestJS services, dependency injection

    ### 3. Domain Layer

    - **Responsibility**: Core business rules and entities
    - **Components**: Entities, domain services, value objects
    - **Technologies**: TypeORM entities, domain models

    ### 4. Infrastructure Layer

    - **Responsibility**: External integrations and persistence
    - **Components**: Repositories, external services, databases
    - **Technologies**: TypeORM repositories, HTTP clients

Step 2.3: Document Key Mechanisms​

  1. Logging Mechanism

    // docs/architecture/mechanisms/logging.md

    ## Logging Strategy

    ### Implementation
    ```typescript
    // Backend Logging (NestJS)
    import { Logger } from '@nestjs/common';

    @Injectable()
    export class LoggingService {
      private readonly logger = new Logger(LoggingService.name);

      logRequest(req: Request, correlationId: string) {
        this.logger.log({
          message: 'HTTP Request',
          method: req.method,
          url: req.url,
          correlationId,
          timestamp: new Date().toISOString()
        });
      }

      logError(error: Error, context: string, correlationId: string) {
        this.logger.error({
          message: error.message,
          stack: error.stack,
          context,
          correlationId,
          timestamp: new Date().toISOString()
        });
      }
    }

    // Frontend Logging (Next.js)
    export class ClientLogger {
      static log(level: 'info' | 'warn' | 'error', message: string, data?: any) {
        const logEntry = {
          level,
          message,
          data,
          timestamp: new Date().toISOString(),
          userAgent: navigator.userAgent,
          url: window.location.href
        };

        // Send to logging service
        this.sendToServer(logEntry);
      }
    }

  2. Error Handling Mechanism

    // docs/architecture/mechanisms/error-handling.md

    ## Error Handling Strategy

    ### Global Error Filter (NestJS)
    ```typescript
    @Catch()
    export class GlobalExceptionFilter implements ExceptionFilter {
      catch(exception: unknown, host: ArgumentsHost) {
        const ctx = host.switchToHttp();
        const response = ctx.getResponse<Response>();
        const request = ctx.getRequest<Request>();

        let status = HttpStatus.INTERNAL_SERVER_ERROR;
        let message = 'Internal server error';

        if (exception instanceof HttpException) {
          status = exception.getStatus();
          message = exception.message;
        }

        const errorResponse = {
          statusCode: status,
          timestamp: new Date().toISOString(),
          path: request.url,
          message,
          correlationId: request.headers['x-correlation-id']
        };

        this.logger.error('Exception caught', exception, request.url);
        response.status(status).json(errorResponse);
      }
    }

    // Error Boundary (Next.js)
    export class ErrorBoundary extends Component {
      constructor(props) {
        super(props);
        this.state = { hasError: false, error: null };
      }

      static getDerivedStateFromError(error) {
        return { hasError: true, error };
      }

      componentDidCatch(error, errorInfo) {
        ClientLogger.log('error', 'React Error Boundary', {
          error: error.message,
          errorInfo
        });
      }
    }

  3. Security Mechanisms

    // docs/architecture/mechanisms/security.md

    ## Security Implementation

    ### Authentication & Authorization
    ```typescript
    // JWT Strategy (NestJS)
    @Injectable()
    export class JwtStrategy extends PassportStrategy(Strategy) {
      constructor() {
        super({
          jwtFromRequest: ExtractJwt.fromAuthHeaderAsBearerToken(),
          ignoreExpiration: false,
          secretOrKey: process.env.JWT_SECRET,
        });
      }

      async validate(payload: any) {
        return {
          userId: payload.sub,
          username: payload.username,
          roles: payload.roles
        };
      }
    }

    // Role-based Guard
    @Injectable()
    export class RolesGuard implements CanActivate {
      constructor(private reflector: Reflector) {}

      canActivate(context: ExecutionContext): boolean {
        const requiredRoles = this.reflector.getAllAndOverride<Role[]>(
          ROLES_KEY,
          [context.getHandler(), context.getClass()]
        );

        if (!requiredRoles) return true;

        const { user } = context.switchToHttp().getRequest();
        return requiredRoles.some((role) => user.roles?.includes(role));
      }
    }

    // Frontend Auth Context (Next.js)
    export const AuthContext = createContext<AuthContextValue | null>(null);

    export function AuthProvider({ children }: { children: ReactNode }) {
      const [user, setUser] = useState<User | null>(null);
      const [loading, setLoading] = useState(true);

      useEffect(() => {
        checkAuthStatus();
      }, []);

      const login = async (credentials: LoginCredentials) => {
        const response = await authAPI.login(credentials);
        setUser(response.user);
        localStorage.setItem('token', response.token);
      };

      return (
        <AuthContext.Provider value={{ user, login, logout, loading }}>
          {children}
        </AuthContext.Provider>
      );
    }

Step 4.2: Create Implementation Guidelines​

Actions:

  1. Development Guidelines Document

    # docs/architecture/implementation-guidelines.md

    ## Architecture Implementation Guidelines

    ### General Principles

    1. **Follow Layer Boundaries**

       - Controllers should only handle HTTP concerns
       - Services contain business logic
       - Repositories handle data access
       - Never skip layers or create circular dependencies

    2. **Dependency Injection**

       - Use constructor injection for dependencies
       - Inject interfaces, not concrete implementations
       - Register all services in appropriate modules

    3. **Error Handling**
       - Use appropriate HTTP status codes
       - Implement correlation IDs for request tracing
       - Log all errors with sufficient context
       - Return consistent error response format

    ### Naming Conventions

    #### Backend (NestJS)

    ```typescript
    // Files and Classes
    user.controller.ts     -> UserController
    user.service.ts        -> UserService
    user.repository.ts     -> UserRepository
    user.entity.ts         -> User
    create-user.dto.ts     -> CreateUserDto

    // Methods
    createUser()           // Verb + noun
    findUserById()         // Action description
    validateUserInput()    // Clear intent
    ```

    Frontend (Next.js)​

    // Files and Components
    user-profile.tsx       -> UserProfile
    user-settings.tsx      -> UserSettings
    api/users/route.ts     -> GET, POST handlers

    // Hooks and Utils
    use-user-data.ts       -> useUserData
    format-date.ts         -> formatDate
    validate-email.ts      -> validateEmail

    # ADR-001: Application Architecture Foundation

    ## Status

    Accepted

    ## Context

    We need to establish a solid architectural foundation for our e-commerce platform that will support scalability, maintainability, and team productivity.

    ## Decision

    We will implement a layered architecture with:

    - NestJS for backend microservices
    - Next.js for frontend microfrontends
    - PostgreSQL for primary data storage
    - Redis for caching and sessions
    - TypeScript throughout the stack

    ## Consequences

    ### Positive

    - Strong typing reduces runtime errors
    - Layered architecture improves maintainability
    - Microservices enable independent scaling
    - Modern frameworks provide extensive tooling

    ### Negative

    - Increased complexity compared to monolithic approach
    - Learning curve for team members
    - Additional operational overhead for multiple services

    ## Implementation

    - Follow the implementation guidelines in this document
    - Use provided templates for consistency
    - Regular architecture reviews to ensure compliance
