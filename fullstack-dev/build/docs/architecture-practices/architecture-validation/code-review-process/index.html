Code Review Process

This guide establishes a comprehensive code review process for NestJS microservices and Next.js microfrontend development teams, focusing on quality, knowledge sharing, and continuous improvement.

Overview​

Code reviews are essential for maintaining code quality, sharing knowledge, catching bugs early, and ensuring adherence to architectural standards. This process is specifically designed for distributed microservices and microfrontend architectures.

Review Process Framework​

Review Types​

Standard Code Reviews​

Pull Request Reviews:

  • All code changes must go through peer review
  • No direct commits to main/develop branches
  • Minimum of 2 approvals required for production code
  • Automated checks must pass before review

Review Stages:

graph TD
    A[Create PR] --> B[Automated Checks]
    B --> C{Checks Pass?}
    C -->|No| D[Fix Issues]
    D --> A
    C -->|Yes| E[Assign Reviewers]
    E --> F[Peer Review]
    F --> G{Approved?}
    G -->|No| H[Address Feedback]
    H --> F
    G -->|Yes| I[Final Approval]
    I --> J[Merge to Target Branch]

Architecture Reviews​

High-Impact Changes:

  • New service creation
  • API contract modifications
  • Database schema changes
  • Cross-service integration changes
  • Security implementation changes

Review Committee:

  • Senior developers from affected teams
  • Architecture team representative
  • DevOps/Platform team member
  • Product owner (for business logic changes)

Automated Review Tools​

GitHub Actions Integration​

# .github/workflows/code-review.yml
name: Automated Code Review

on:
  pull_request:
    branches: [main, develop]

jobs:
  automated-review:
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run ESLint
        run: npm run lint -- --format=json --output-file=eslint-report.json
        continue-on-error: true

      - name: Run Tests with Coverage
        run: npm run test:coverage -- --coverage --coverageReporters=json
        continue-on-error: true

      - name: Security Audit
        run: npm audit --json > security-audit.json
        continue-on-error: true

      - name: Type Check
        run: npm run type-check

      - name: SonarCloud Scan
        uses: SonarSource/sonarcloud-github-action@master
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}

      - name: Comment PR with Review
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            
            // Read analysis results
            const eslintReport = JSON.parse(fs.readFileSync('eslint-report.json', 'utf8'));
            const coverageReport = JSON.parse(fs.readFileSync('coverage/coverage-summary.json', 'utf8'));
            
            // Generate review comment
            const comment = `## Automated Code Review
            
            ### ESLint Results
            - Total issues: ${eslintReport.reduce((acc, file) => acc + file.errorCount + file.warningCount, 0)}
            - Errors: ${eslintReport.reduce((acc, file) => acc + file.errorCount, 0)}
            - Warnings: ${eslintReport.reduce((acc, file) => acc + file.warningCount, 0)}
            
            ### Test Coverage
            - Lines: ${coverageReport.total.lines.pct}%
            - Functions: ${coverageReport.total.functions.pct}%
            - Branches: ${coverageReport.total.branches.pct}%
            - Statements: ${coverageReport.total.statements.pct}%
            
            ### Recommendations
            ${eslintReport.reduce((acc, file) => acc + file.errorCount, 0) > 0 ? '- ❌ Fix ESLint errors before merge' : '- ✅ No ESLint errors'}
            ${coverageReport.total.lines.pct < 80 ? '- ❌ Increase test coverage above 80%' : '- ✅ Test coverage looks good'}
            `;
            
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: comment
            });

Code Quality Bot Integration​

// scripts/review-bot.ts
import { Octokit } from '@octokit/rest';
import { PullRequest, Review } from '@octokit/webhooks-types';

class CodeReviewBot {
  private octokit: Octokit;
  
  constructor(token: string) {
    this.octokit = new Octokit({ auth: token });
  }
  
  async analyzePullRequest(pr: PullRequest): Promise<void> {
    const analysis = await this.performCodeAnalysis(pr);
    const review = this.generateReview(analysis);
    
    await this.postReview(pr, review);
  }
  
  private async performCodeAnalysis(pr: PullRequest): Promise<AnalysisResult> {
    const files = await this.octokit.pulls.listFiles({
      owner: pr.base.repo.owner.login,
      repo: pr.base.repo.name,
      pull_number: pr.number,
    });
    
    return {
      complexity: this.analyzeComplexity(files.data),
      patterns: this.analyzePatterns(files.data),
      security: this.analyzeSecurityIssues(files.data),
      performance: this.analyzePerformanceIssues(files.data),
    };
  }
  
  private generateReview(analysis: AnalysisResult): ReviewComment[] {
    const comments: ReviewComment[] = [];
    
    // Generate specific line comments based on analysis
    if (analysis.complexity.highComplexityFunctions.length > 0) {
      comments.push({
        path: analysis.complexity.highComplexityFunctions[0].file,
        line: analysis.complexity.highComplexityFunctions[0].line,
        body: "Consider breaking down this function - it has high cyclomatic complexity.",
      });
    }
    
    if (analysis.security.vulnerabilities.length > 0) {
      analysis.security.vulnerabilities.forEach(vuln => {
        comments.push({
          path: vuln.file,
          line: vuln.line,
          body: `Security issue detected: ${vuln.description}`,
        });
      });
    }
    
    return comments;
  }
}

class ReviewMetricsCollector {
  async collectWeeklyMetrics(): Promise<ReviewMetrics> {
    const prs = await this.getPullRequestsFromLastWeek();
    const reviews = await this.getReviewsForPRs(prs);
    
    return {
      reviewTimes: this.calculateReviewTimes(prs, reviews),
      qualityIndicators: this.calculateQualityIndicators(prs),
      participationMetrics: this.calculateParticipationMetrics(reviews),
      codeQualityTrends: this.calculateQualityTrends(prs),
    };
  }
}

Best Practices​

Review Efficiency​

  1. Time Boxing: Limit review sessions to 60-90 minutes
  2. Focus Areas: Review no more than 400 lines of code at once
  3. Clear Objectives: Know what to look for in each review type
  4. Tool Usage: Leverage automated tools for routine checks
  5. Documentation: Provide clear, actionable feedback

Knowledge Sharing​

  1. Learning Opportunities: Use reviews to teach and learn
  2. Pattern Recognition: Share common patterns and anti-patterns
  3. Domain Knowledge: Transfer business logic understanding
  4. Best Practices: Propagate coding standards and conventions
  5. Architecture Awareness: Ensure architectural compliance

Quality Assurance​

  1. Consistency: Apply standards uniformly across all reviews
  2. Completeness: Don't rush through reviews
  3. Constructive Feedback: Focus on improvement, not criticism
  4. Follow-up: Verify that feedback is addressed appropriately
  5. Continuous Improvement: Regularly update review processes
