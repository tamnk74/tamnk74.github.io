GKE Ingress Setup Guide

A comprehensive guide to setting up and configuring ingress controllers in Google Kubernetes Engine (GKE).

Overview​

Ingress in Kubernetes provides HTTP and HTTPS routing to services within your cluster. In GKE, you have several options for ingress controllers, each with their own benefits and use cases.

Prerequisites​

Before setting up ingress, ensure you have:

  • A running GKE cluster
  • kubectl configured to access your cluster
  • gcloud CLI installed and authenticated
  • Domain name pointing to your cluster (for SSL/TLS)
  • Required IAM permissions

Required IAM Permissions​

# Grant necessary permissions for ingress setup
gcloud projects add-iam-policy-binding PROJECT_ID \
    --member="serviceAccount:SERVICE_ACCOUNT_EMAIL" \
    --role="roles/compute.loadBalancerAdmin"

gcloud projects add-iam-policy-binding PROJECT_ID \
    --member="serviceAccount:SERVICE_ACCOUNT_EMAIL" \
    --role="roles/compute.securityAdmin"

Ingress Controller Options​

1. GKE Ingress (Google Cloud Load Balancer)​

  • Pros: Native integration, automatic SSL certificates, global load balancing
  • Cons: Limited customization, Google Cloud specific
  • Best for: Simple setups, Google Cloud native solutions

2. NGINX Ingress Controller​

  • Pros: Highly customizable, portable across clouds, rich feature set
  • Cons: Additional management overhead, requires more configuration
  • Best for: Complex routing requirements, multi-cloud deployments

GKE Ingress (Google Cloud Load Balancer)​

Step 1: Enable Required APIs​

# Enable necessary Google Cloud APIs
gcloud services enable compute.googleapis.com
gcloud services enable container.googleapis.com
gcloud services enable certificatemanager.googleapis.com

Step 2: Create a Sample Application​

Create a deployment and service:

# app-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  labels:
    app: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
    spec:
      containers:
      - name: web-app
        image: nginx:1.21
        ports:
        - containerPort: 80
        resources:
          requests:
            memory: "64Mi"
            cpu: "250m"
          limits:
            memory: "128Mi"
            cpu: "500m"
---
apiVersion: v1
kind: Service
metadata:
  name: web-app-service
  annotations:
    cloud.google.com/neg: '{"ingress": true}'
spec:
  type: ClusterIP
  selector:
    app: web-app
  ports:
  - port: 80
    targetPort: 80
    protocol: TCP

Apply the configuration:

kubectl apply -f app-deployment.yaml

Step 3: Create GKE Ingress​

# gke-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: web-app-ingress
  annotations:
    kubernetes.io/ingress.class: "gce"
    kubernetes.io/ingress.global-static-ip-name: "web-app-ip"
    networking.gke.io/managed-certificates: "web-app-ssl-cert"
    kubernetes.io/ingress.allow-http: "false"
spec:
  rules:
  - host: your-domain.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: web-app-service
            port:
              number: 80

Step 4: Reserve Static IP​

# Reserve a global static IP
gcloud compute addresses create web-app-ip --global

# Get the IP address
gcloud compute addresses describe web-app-ip --global --format="value(address)"

Step 5: Create Managed SSL Certificate​

# managed-cert.yaml
apiVersion: networking.gke.io/v1
kind: ManagedCertificate
metadata:
  name: web-app-ssl-cert
spec:
  domains:
    - your-domain.com
    - www.your-domain.com

Apply the configurations:

kubectl apply -f managed-cert.yaml
kubectl apply -f gke-ingress.yaml

NGINX Ingress Controller​

Step 1: Install NGINX Ingress Controller​

Prerequisites for GKE​

First, ensure your user has the necessary permissions:

# Grant cluster-admin permissions (required for NGINX installation)
kubectl create clusterrolebinding cluster-admin-binding \
  --clusterrole cluster-admin \
  --user $(gcloud config get-value account)

Installation Methods​

Option A: Using Helm (Recommended)

# Add NGINX Ingress Helm repository
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update

# Install NGINX Ingress Controller with GKE-optimized settings
helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  --create-namespace \
  --set controller.service.type=LoadBalancer \
  --set controller.service.externalTrafficPolicy=Local \
  --set controller.config.use-proxy-protocol=false \
  --set controller.service.annotations."cloud\.google\.com/load-balancer-type"="External" \
  --set controller.service.annotations."cloud\.google\.com/neg"='{"ingress": true}' \
  --set controller.metrics.enabled=true \
  --set controller.podSecurityContext.runAsNonRoot=true \
  --set controller.podSecurityContext.runAsUser=101 \
  --set controller.podSecurityContext.fsGroup=101

Option B: Using kubectl with YAML manifest

# Install using the official GKE-compatible manifest
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.13.3/deploy/static/provider/cloud/deploy.yaml

GKE-Specific Configuration Considerations​

For Private GKE Clusters:

Private clusters require additional firewall rules for the admission webhook:

# Add firewall rule for private clusters (webhook communication)
gcloud compute firewall-rules create allow-ingress-nginx-admission \
  --direction=INGRESS \
  --priority=1000 \
  --network=default \
  --action=ALLOW \
  --rules=tcp:8443 \
  --source-ranges=172.16.0.0/12 \
  --target-tags=gke-node

# Replace source-ranges with your cluster's master CIDR
# You can find this with: gcloud container clusters describe CLUSTER_NAME --zone=ZONE

Verification​

# Wait for the controller to be ready
kubectl wait --namespace ingress-nginx \
  --for=condition=ready pod \
  --selector=app.kubernetes.io/component=controller \
  --timeout=120s

# Check the LoadBalancer service
kubectl get service ingress-nginx-controller -n ingress-nginx

# Expected output should show an EXTERNAL-IP
# NAME                                 TYPE           CLUSTER-IP     EXTERNAL-IP     PORT(S)
# ingress-nginx-controller             LoadBalancer   10.48.15.200   34.123.45.67    80:32000/TCP,443:32001/TCP

How NGINX Ingress Creates Load Balancers​

When you install NGINX Ingress Controller with controller.service.type=LoadBalancer, it automatically creates a Google Cloud Load Balancer. Here's how the process works:

1. Service Creation Process:

The NGINX controller creates a Kubernetes Service of type LoadBalancer, which triggers GKE to provision a Google Cloud Load Balancer automatically.

2. Load Balancer Components Created:

When the NGINX controller starts, GKE automatically creates these GCP resources:

  • External Load Balancer: Routes traffic from internet to NGINX pods
  • Backend Service: Manages health checks and traffic distribution
  • Target Pool/Instance Groups: Contains GKE nodes running NGINX pods
  • Firewall Rules: Allow traffic on ports 80 and 443
  • Health Checks: Monitor NGINX controller health on port 10254

3. Verify Load Balancer Creation:

# Get the external IP address
EXTERNAL_IP=$(kubectl get service ingress-nginx-controller \
  -n ingress-nginx \
  -o jsonpath='{.status.loadBalancer.ingress[0].ip}')

echo "Load Balancer External IP: $EXTERNAL_IP"

# Test load balancer connectivity
curl -k https://$EXTERNAL_IP/healthz

# Check load balancer components in GCP
gcloud compute forwarding-rules list --filter="name~k8s"
gcloud compute backend-services list --filter="name~k8s"
gcloud compute health-checks list --filter="name~k8s"

Step 2: Create IngressClass (Recommended Approach)​

Modern Kubernetes (1.19+) uses IngressClass instead of the deprecated annotation:

# nginx-ingress-class.yaml
apiVersion: networking.k8s.io/v1
kind: IngressClass
metadata:
  name: nginx
  annotations:
    ingressclass.kubernetes.io/is-default-class: "true"
spec:
  controller: k8s.io/ingress-nginx

Apply the IngressClass:

kubectl apply -f nginx-ingress-class.yaml

Advanced Load Balancer Configuration​

1. Internal Load Balancer for NGINX:

If you need an internal load balancer (only accessible within VPC):

# Install NGINX with internal load balancer
helm install ingress-nginx-internal ingress-nginx/ingress-nginx \
  --namespace ingress-nginx-internal \
  --create-namespace \
  --set controller.service.type=LoadBalancer \
  --set controller.service.annotations."cloud\.google\.com/load-balancer-type"="Internal" \
  --set controller.service.annotations."cloud\.google\.com/load-balancer-subnet"="internal-subnet" \
  --set controller.ingressClass="nginx-internal" \
  --set controller.ingressClassResource.name="nginx-internal"

2. Regional Load Balancer Configuration:

# Install with optimized regional load balancer
helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  --create-namespace \
  --set controller.service.type=LoadBalancer \
  --set controller.service.externalTrafficPolicy=Local \
  --set controller.service.annotations."cloud\.google\.com/load-balancer-type"="External" \
  --set controller.service.annotations."cloud\.google\.com/neg"='{"ingress": true}' \
  --set controller.service.annotations."cloud\.google\.com/backend-config"='{"default": "nginx-backend-config"}'

3. Static IP Assignment:

# Reserve a global static IP for the load balancer
gcloud compute addresses create nginx-static-ip --global

# Get the reserved IP
STATIC_IP=$(gcloud compute addresses describe nginx-static-ip --global --format="value(address)")

# Install NGINX with static IP
helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx \
  --create-namespace \
  --set controller.service.type=LoadBalancer \
  --set controller.service.loadBalancerIP="$STATIC_IP" \
  --set controller.service.annotations."cloud\.google\.com/load-balancer-type"="External"

Load Balancer Service Configuration Options​

1. Complete Service Configuration:

# nginx-lb-service-modern.yaml
apiVersion: v1
kind: Service
metadata:
  name: ingress-nginx-controller
  namespace: ingress-nginx
  annotations:
    # Load balancer type (External or Internal)
    cloud.google.com/load-balancer-type: "External"
    
    # Enable NEG for better performance and health checks
    cloud.google.com/neg: '{"ingress": true}'
    
    # Static IP (optional)
    # cloud.google.com/load-balancer-ip: "34.123.45.67"
    
    # Backend configuration
    cloud.google.com/backend-config: '{"default": "nginx-backend-config"}'
    
    # Load balancer name (optional, for easier identification)
    cloud.google.com/load-balancer-name: "nginx-ingress-lb"
spec:
  type: LoadBalancer
  externalTrafficPolicy: Local  # Preserves source IP and reduces latency
  sessionAffinity: None
  ipFamilyPolicy: SingleStack
  ipFamilies:
  - IPv4
  ports:
  - name: http
    port: 80
    protocol: TCP
    targetPort: http
    appProtocol: http
  - name: https
    port: 443
    protocol: TCP
    targetPort: https
    appProtocol: https
  selector:
    app.kubernetes.io/component: controller
    app.kubernetes.io/instance: ingress-nginx
    app.kubernetes.io/name: ingress-nginx

NGINX Ingress Annotations Explained​

The annotations in the above example control how NGINX processes requests. Here's a detailed explanation of each annotation and many other commonly used ones:

Core Annotations (from the example)​

kubernetes.io/ingress.class: "nginx"

  • Purpose: Specifies which ingress controller should handle this ingress
  • Usage: Tells Kubernetes to use the NGINX Ingress Controller
  • Example: kubernetes.io/ingress.class: "nginx"
  • Note: In newer Kubernetes versions, use spec.ingressClassName: nginx instead

nginx.ingress.kubernetes.io/rewrite-target: /

  • Purpose: Rewrites the request path before forwarding to the backend
  • Usage: Removes path prefixes or transforms URLs
  • Example: Request to /api/v1/users becomes /users if path is /api/v1 and rewrite-target is /
  • Advanced: Use capture groups: nginx.ingress.kubernetes.io/rewrite-target: /$2 with path /something(/|$)(.*)

nginx.ingress.kubernetes.io/ssl-redirect: "true"

  • Purpose: Automatically redirects HTTP requests to HTTPS
  • Usage: Forces secure connections for all traffic
  • Values: "true" (redirect) or "false" (allow HTTP)
  • Default: Depends on ingress controller configuration

cert-manager.io/cluster-issuer: "letsencrypt-prod"

  • Purpose: Automatically provisions SSL certificates using cert-manager
  • Usage: References a ClusterIssuer resource for certificate generation
  • Example: Uses Let's Encrypt production environment for certificates
  • Alternative: cert-manager.io/issuer: "my-issuer" for namespace-scoped issuer

Traffic Management Annotations​

# Complete traffic management example
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: traffic-management-ingress
  annotations:
    # Load balancing configuration
    nginx.ingress.kubernetes.io/upstream-hash-by: "$request_uri"
    nginx.ingress.kubernetes.io/load-balance: "ip_hash"
    
    # Session affinity
    nginx.ingress.kubernetes.io/affinity: "cookie"
    nginx.ingress.kubernetes.io/affinity-mode: "persistent"
    nginx.ingress.kubernetes.io/session-cookie-name: "NGINX_AFFINITY"
    nginx.ingress.kubernetes.io/session-cookie-expires: "86400"
    nginx.ingress.kubernetes.io/session-cookie-max-age: "86400"
    nginx.ingress.kubernetes.io/session-cookie-path: "/"
    
    # Rate limiting
    nginx.ingress.kubernetes.io/rate-limit: "100"
    nginx.ingress.kubernetes.io/rate-limit-window: "1m"
    nginx.ingress.kubernetes.io/rate-limit-connections: "10"

Load Balancing Annotations:

  • nginx.ingress.kubernetes.io/load-balance: Algorithm for distributing requests

    • Values: round_robin (default), least_conn, ip_hash, random
    • Example: ip_hash ensures same client always goes to same backend

  • nginx.ingress.kubernetes.io/upstream-hash-by: Custom hash key for load balancing

    • Values: "$request_uri", "$remote_addr", "$http_x_forwarded_for"
    • Example: Hash by URI to ensure same endpoint for same request

Session Affinity:

  • nginx.ingress.kubernetes.io/affinity: "cookie": Enable session persistence
  • nginx.ingress.kubernetes.io/session-cookie-name: Cookie name for affinity
  • nginx.ingress.kubernetes.io/session-cookie-expires: Cookie expiration time

Rate Limiting:

  • nginx.ingress.kubernetes.io/rate-limit: Requests per minute per IP
  • nginx.ingress.kubernetes.io/rate-limit-window: Time window for rate limiting
  • nginx.ingress.kubernetes.io/rate-limit-connections: Concurrent connections per IP

Security Annotations​

# Security-focused ingress
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: security-ingress
  annotations:
    # CORS configuration
    nginx.ingress.kubernetes.io/enable-cors: "true"
    nginx.ingress.kubernetes.io/cors-allow-origin: "https://trusted-domain.com"
    nginx.ingress.kubernetes.io/cors-allow-methods: "GET, POST, PUT, DELETE"
    nginx.ingress.kubernetes.io/cors-allow-headers: "Authorization, Content-Type"
    nginx.ingress.kubernetes.io/cors-allow-credentials: "true"
    
    # Authentication
    nginx.ingress.kubernetes.io/auth-type: "basic"
    nginx.ingress.kubernetes.io/auth-secret: "basic-auth-secret"
    nginx.ingress.kubernetes.io/auth-realm: "Authentication Required"
    
    # Security headers
    nginx.ingress.kubernetes.io/server-snippet: |
      add_header X-Frame-Options "SAMEORIGIN" always;
      add_header X-Content-Type-Options "nosniff" always;
      add_header X-XSS-Protection "1; mode=block" always;
      add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    
    # Client certificate authentication
    nginx.ingress.kubernetes.io/auth-tls-verify-client: "on"

Best Practices for Internal Load Balancers​

  1. Network Security:

    # Restrict access to specific subnets
    gcloud compute firewall-rules create restrict-internal-access \
        --direction=INGRESS \
        --action=ALLOW \
        --rules=tcp:80,tcp:443 \
        --source-ranges=10.10.0.0/16 \
        --target-tags=internal-lb-restricted

    Command Explanation:

    This firewall rule creates a security boundary for internal load balancers by restricting network access to specific IP ranges. Let's break down each parameter:

    • gcloud compute firewall-rules create restrict-internal-access

      • Creates a new firewall rule named "restrict-internal-access"
      • This rule will be applied at the VPC network level

    • --direction=INGRESS

      • Specifies this rule applies to incoming traffic (traffic entering your network)
      • Alternative would be EGRESS for outgoing traffic

    • --action=ALLOW

      • Permits the specified traffic to pass through
      • Alternative would be DENY to block traffic

    • --rules=tcp:80,tcp:443

      • Allows TCP traffic on port 80 (HTTP) and port 443 (HTTPS)
      • Format: protocol:port or protocol:port-range
      • Examples: tcp:8080, tcp:8000-9000, udp:53

    • --source-ranges=10.10.0.0/16

      • Restricts access to only IP addresses in the 10.10.0.0/16 CIDR block
      • This means only IPs from 10.10.0.1 to 10.10.255.254 can access the service
      • Common private IP ranges:
        • 10.0.0.0/8 (10.0.0.0 - 10.255.255.255)
        • 172.16.0.0/12 (172.16.0.0 - 172.31.255.255)
        • 192.168.0.0/16 (192.168.0.0 - 192.168.255.255)

    • --target-tags=internal-lb-restricted

      • Applies this rule only to resources tagged with internal-lb-restricted
      • Resources must have this network tag to be affected by this firewall rule
      • You can assign tags to VMs, load balancers, and other compute resources

    Real-World Use Cases:

    # Example 1: Restrict to office network only
    gcloud compute firewall-rules create office-only-access \
        --direction=INGRESS \
        --action=ALLOW \
        --rules=tcp:80,tcp:443 \
        --source-ranges=203.0.113.0/24 \
        --target-tags=office-accessible

    # Example 2: Allow access from multiple internal subnets
    gcloud compute firewall-rules create multi-subnet-access \
        --direction=INGRESS \
        --action=ALLOW \
        --rules=tcp:80,tcp:443,tcp:8080 \
        --source-ranges=10.1.0.0/16,10.2.0.0/16,10.3.0.0/16 \
        --target-tags=internal-services

    # Example 3: Restrict database access to app servers only
    gcloud compute firewall-rules create db-access-restricted \
        --direction=INGRESS \
        --action=ALLOW \
        --rules=tcp:5432,tcp:3306 \
        --source-ranges=10.100.1.0/24 \
        --target-tags=database-servers

    Security Benefits:

    • ✅ Zero-trust networking: Only specified IP ranges can access services
    • ✅ Defense in depth: Additional layer beyond application-level security
    • ✅ Compliance: Meets regulatory requirements for network isolation
    • ✅ Audit trail: All firewall rule changes are logged for security auditing

5. Backup and Disaster Recovery​

Implementing a comprehensive backup and disaster recovery strategy for your GKE ingress setup is crucial for maintaining high availability and minimizing downtime during unexpected events.

5.1 Configuration Backup Strategies​

Automated Configuration Export​

Create automated scripts to backup your ingress configurations:

#!/bin/bash
# backup-ingress-configs.sh

# Set variables
BACKUP_DIR="/backup/ingress-configs/$(date +%Y-%m-%d)"
NAMESPACE="default"

# Create backup directory
mkdir -p "$BACKUP_DIR"

# Backup all ingress resources
echo "Backing up ingress configurations..."
kubectl get ingress -n "$NAMESPACE" -o yaml > "$BACKUP_DIR/ingress-backup.yaml"

# Backup services
kubectl get services -n "$NAMESPACE" -o yaml > "$BACKUP_DIR/services-backup.yaml"

# Backup SSL certificates (managed certificates)
kubectl get managedcertificate -n "$NAMESPACE" -o yaml > "$BACKUP_DIR/managed-certs-backup.yaml"

# Backup cert-manager certificates (if using NGINX)
kubectl get certificates -n "$NAMESPACE" -o yaml > "$BACKUP_DIR/certificates-backup.yaml"

# Backup secrets (TLS certificates)
kubectl get secrets -n "$NAMESPACE" --field-selector type=kubernetes.io/tls -o yaml > "$BACKUP_DIR/tls-secrets-backup.yaml"

# Backup backend configs (GKE specific)
kubectl get backendconfig -n "$NAMESPACE" -o yaml > "$BACKUP_DIR/backend-configs-backup.yaml"

# Backup ingress controller configuration (NGINX)
if kubectl get namespace ingress-nginx &> /dev/null; then
    kubectl get configmap -n ingress-nginx -o yaml > "$BACKUP_DIR/nginx-configmap-backup.yaml"
    kubectl get deployment -n ingress-nginx -o yaml > "$BACKUP_DIR/nginx-deployment-backup.yaml"
fi

# Create tarball
tar -czf "$BACKUP_DIR/../ingress-backup-$(date +%Y-%m-%d).tar.gz" -C "$BACKUP_DIR/.." "$(basename "$BACKUP_DIR")"

echo "Backup completed: $BACKUP_DIR/../ingress-backup-$(date +%Y-%m-%d).tar.gz"

Store configurations in Git for version control and easy restoration:

# Initialize git repository for configurations

#!/bin/bash
# test-failover.sh

# Configuration
PRIMARY_ENDPOINT="https://your-domain.com"
SECONDARY_ENDPOINT="https://backup.your-domain.com"
HEALTH_CHECK_PATH="/health"

echo "Starting failover test..."

echo "Testing primary endpoint: $PRIMARY_ENDPOINT"
primary_status=$(curl -s -o /dev/null -w "%{http_code}" "$PRIMARY_ENDPOINT$HEALTH_CHECK_PATH")

if [ "$primary_status" -eq 200 ]; then
    echo "✅ Primary endpoint is healthy (Status: $primary_status)"
else
    echo "❌ Primary endpoint failed (Status: $primary_status)"
    
    # Test secondary endpoint
    echo "Testing secondary endpoint: $SECONDARY_ENDPOINT"
    secondary_status=$(curl -s -o /dev/null -w "%{http_code}" "$SECONDARY_ENDPOINT$HEALTH_CHECK_PATH")
    
    if [ "$secondary_status" -eq 200 ]; then
        echo "✅ Secondary endpoint is healthy (Status: $secondary_status)"
        echo "🔄 Initiating failover procedures..."
        
        # Add your failover logic here
        # Examples:
        # - Update DNS records
        # - Notify monitoring systems
        # - Scale up secondary region
        
    else
        echo "❌ Both endpoints failed! Manual intervention required."
        # Send critical alerts
    fi
fi

# Test SSL certificate expiration
echo "Checking SSL certificate expiration..."
cert_expiry=$(echo | openssl s_client -servername your-domain.com -connect your-domain.com:443 2>/dev/null | openssl x509 -noout -dates | grep "notAfter" | cut -d= -f2)
expiry_epoch=$(date -d "$cert_expiry" +%s)
current_epoch=$(date +%s)
days_until_expiry=$(( (expiry_epoch - current_epoch) / 86400 ))

if [ "$days_until_expiry" -lt 30 ]; then
    echo "⚠️  SSL certificate expires in $days_until_expiry days"
else
    echo "✅ SSL certificate is valid for $days_until_expiry days"
fi

5.4 Recovery Procedures​

Ingress Configuration Recovery​
#!/bin/bash
# restore-ingress.sh

BACKUP_FILE="$1"
NAMESPACE="${2:-default}"

if [ -z "$BACKUP_FILE" ]; then
    echo "Usage: $0 <backup-file> [namespace]"
    exit 1
fi

echo "Restoring ingress configuration from: $BACKUP_FILE"

# Extract backup if it's a tarball
if [[ "$BACKUP_FILE" == *.tar.gz ]]; then
    tar -xzf "$BACKUP_FILE"
    BACKUP_DIR=$(basename "$BACKUP_FILE" .tar.gz)
else
    BACKUP_DIR="$BACKUP_FILE"
fi

# Restore ingress resources
echo "Restoring ingress resources..."
kubectl apply -f "$BACKUP_DIR/ingress-backup.yaml" -n "$NAMESPACE"

# Restore services
echo "Restoring services..."
kubectl apply -f "$BACKUP_DIR/services-backup.yaml" -n "$NAMESPACE"

# Restore managed certificates
echo "Restoring managed certificates..."
if [ -f "$BACKUP_DIR/managed-certs-backup.yaml" ]; then
    kubectl apply -f "$BACKUP_DIR/managed-certs-backup.yaml" -n "$NAMESPACE"
fi

# Restore TLS secrets
echo "Restoring TLS secrets..."
if [ -f "$BACKUP_DIR/tls-secrets-backup.yaml" ]; then
    kubectl apply -f "$BACKUP_DIR/tls-secrets-backup.yaml" -n "$NAMESPACE"
fi

# Restore backend configs
echo "Restoring backend configurations..."
if [ -f "$BACKUP_DIR/backend-configs-backup.yaml" ]; then
    kubectl apply -f "$BACKUP_DIR/backend-configs-backup.yaml" -n "$NAMESPACE"
fi

echo "Recovery completed. Verifying ingress status..."
kubectl get ingress -n "$NAMESPACE"

echo "Restoring DNS records..."
ZONES=$(gcloud dns managed-zones list --project="$PROJECT_ID" --format="value(name)")

for zone in $ZONES; do

        echo "Importing records from backup file"
    fi
done

Automated Recovery Triggers​
#!/bin/bash
# automated-recovery.sh

# This script can be triggered by monitoring alerts
ALERT_TYPE="$1"
SEVERITY="$2"

case "$ALERT_TYPE" in
    "ingress_down")
        echo "Ingress is down. Starting recovery procedures..."
        # Restart ingress controllers
        kubectl rollout restart deployment/ingress-nginx-controller -n ingress-nginx
        ;;
    "ssl_expiring")
        echo "SSL certificate expiring. Triggering renewal..."
        # Force certificate renewal
        kubectl annotate managedcertificate web-app-ssl-cert cert-manager.io/force-renewal=$(date +%s)
        ;;
    "high_error_rate")
        echo "High error rate detected. Scaling ingress controllers..."
        # Scale ingress controllers
        kubectl scale deployment/ingress-nginx-controller --replicas=5 -n ingress-nginx
        ;;
    *)
        echo "Unknown alert type: $ALERT_TYPE"
        ;;
esac

Disaster Recovery Runbook​

Create a detailed runbook for your team:

# Ingress Disaster Recovery Runbook

## Emergency Contacts
- DevOps Team Lead: +1-xxx-xxx-xxxx
- Security Team: security@company.com
- DNS Provider Support: dns-support@provider.com

## Recovery Procedures by Scenario

### Scenario 1: Complete Ingress Failure
1. Check ingress controller status: `kubectl get pods -n ingress-nginx`
2. Check load balancer status in GCP Console
3. Verify DNS resolution: `dig your-domain.com`
4. If needed, restore from backup: `./restore-ingress.sh backup-file.tar.gz`

### Scenario 2: SSL Certificate Issues
1. Check certificate status: `kubectl get managedcertificate`
2. Verify DNS records point to correct IP
3. Check certificate expiration
4. Force renewal if needed

### Scenario 3: DNS Resolution Problems
1. Check DNS propagation: `dig +trace your-domain.com`
2. Verify DNS records in provider console
3. Check static IP assignment
4. Restore DNS records if needed: `./restore-dns.sh project-id backup-date`

## Recovery Time Objectives (RTO)
- DNS failover: 5 minutes
- Ingress restoration: 15 minutes
- Full service recovery: 30 minutes

## Recovery Point Objectives (RPO)
- Configuration backups: 4 hours
- DNS record backups: Daily

This comprehensive backup and disaster recovery strategy ensures that your GKE ingress setup can be quickly restored in case of failures, with minimal downtime and data loss.
